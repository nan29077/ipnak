## 0. 한눈에 보기 (요약)

분석한 앱은 760px 중앙 프레임의 모바일‑우선 낚시 커뮤니티로, 피드 · 데이터피싱(GPS) · 스마트 자 · 지도/포인트 · 대회 · 예약 · 중고마켓 · 쇼핑 · 마이 · 관리자 10개 영역으로 구성됩니다. 전체 약 7,400줄, 페이지 37개, 컴포넌트 38개, API 라우트 25개로 기능은 거의 완성형입니다.

성능 관점에서 **가장 효과가 큰 세 가지**는 ① Prisma 스키마에 인덱스가 **하나도 없음**(`@@index` 0개), ② 모든 페이지가 `force-dynamic`이라 캐싱이 전혀 없음, ③ `next/image`를 전혀 쓰지 않아 이미지가 최적화·지연로딩되지 않는 점입니다. 이 셋만 고쳐도 체감 속도가 크게 올라갑니다.

디자인 관점에서는 토큰(`surface`)이 정의돼 있는데도 `#161616`·`#1e1e1e` 같은 **하드코딩 색상이 50회 이상** 흩어져 있고, 동일한 `Chip`·필터 로직이 중복 정의돼 있으며, 빈 `src=""` 이미지로 인한 깨짐/레이아웃 점프가 일관성을 떨어뜨립니다.

## 1. 기능별 화면 플로우 (시작 → 중간 → 끝)

각 플로우는 "사용자가 어느 화면에서 → 어떤 순서로 동작 → 어디서 끝나는지"를 코드 기준으로 정리했습니다.

### 1.1 공통 셸 / 네비게이션
- **구조**: `layout.tsx`(서버) → `getCurrentUser()`로 세션 조회 → `AppShell`이 전체를 감쌈.
- **모바일**: 하단 고정 탭 5개(홈·데이터피싱·대회·예약·쇼핑·마이) + 중앙에 떠 있는 주황 `＋`(잡은 물고기 기록 `/catch/new`).
- **PC/태블릿(md↑)**: 앱 프레임 우측에 세로 패널(브랜드 → 메뉴 → 기록 버튼 → 계정/관리자).
- `/login`, `/signup`, `/admin`은 "bare" 모드로 셸을 숨김.
- 홈 영역 안에서 다시 `CommunityTabs`(피드 ↔ 중고피싱) 세그먼트로 갈라짐.

### 1.2 피드 (홈, `/` · `/feed`)
시작: `HomePage`(서버)가 `getFeedPosts()` + 활성 배너를 조회 → `FeedList`로 전달.
중간: `FeedHeader`(전체/피싱포인트/일반 필터) → `CommunityTabs` → 배너 캐러셀 → `FeedCard` 목록.
`FeedCard` 내 상호작용: 좋아요(낙관적 토글, 더블탭 버스트), 댓글 시트(열 때 `/comments` 지연 로드), 공유(Web Share→클립보드 폴백), 저장(북마크), 쇼핑 태그 핀 토글, 다중 이미지 캐러셀, 신고(더보기).
끝: 카드 탭 → `/post/[id]` 단일 게시글, 또는 작성자 탭 → `/profile/[id]`.

### 1.3 데이터피싱 GPS 기록 (`/map`)
시작: `MapScreen`(클라이언트) 마운트 → `/api/points`(공개 포인트) + `/api/trips`(내 기록) 로드 + 현재 위치 시도.
중간: 상단 검색바·"내 기록" 버튼, 지역 빠른 이동 칩(`KOREA_SPOTS`), Leaflet 지도(`MapView`→동적 `MapCanvas`, SSR off).
기록 흐름: **시작**(`watchPosition`, 권한 없으면 `mockRoute` 시뮬레이션) → 경로/거리/시간 누적 → **일시정지**/**재개** → **종료**.
종료 시: 낙관적으로 로컬 목록에 추가 + 토스트 → `/api/trips` POST로 서버 저장 → 임시 id를 서버 id로 치환.
끝: 저장된 기록을 "피드에 공유"(`/api/posts`) 하거나, 상세 시트(`TripDetailSheet`)로 경로 확인.

### 1.4 스마트 자 계측 (`/ruler` 및 `/catch/new` 내 시트)
시작: `PhotoPicker`로 사진 1장 선택/촬영.
중간: `SmartRuler`에서 ① 기준 자 시작점 → ② 끝점 → ③ 물고기 입 → ④ 꼬리 4점을 탭(드래그 미세조정, 휠/버튼 확대·이동). 기준 길이(10/30/50cm 또는 직접 입력)로 cm 환산, 신뢰도 자동 계산.
끝: "측정 완료" → `RulerResult` 반환(`/ruler`는 JSON 표시, `/catch/new`는 크기 필드 자동 채움). 결과 이미지를 캔버스로 PNG 저장 가능.

### 1.5 잡은 물고기 기록 (`/catch/new`) — 앱의 핵심 작성 플로우
시작: 진입 시 `getCurrentPosition`으로 위치 자동 시도.
중간: 사진(최대 3장) → 측정 결과 카드 → 스마트 자 시트 → 어종/크기 → 메모 → 낚시 분류(바다/민물 + 환경 + 방식) → 채비/장비 → 쇼핑 상품 태그(`ProductTagPicker` + 사진 위 위치 지정 `ProductTagPlacer`) → 위치/포인트 공개 범위 → "피드 공유" 토글 + 공개 범위.
끝: `submit()` → `/api/catch` POST(서버에서 `fishingPoint` + `catchRecord` + 선택 시 `post`까지 한 번에 생성) → 공유 시 `/post/[id]`, 비공개 시 `/trip`으로 이동.

### 1.6 내 낚시 기록 (`/trip`)
시작(서버): 미로그인 시 `/login` 리다이렉트, 로그인 시 `fishingTrip`(경로·포인트 include) 조회.

## 2. 속도 향상 권고 (우선순위 순)

> 우선순위: **P0**=가장 큰 효과·낮은 리스크, **P1**=중간, **P2**=점진 개선.

### P0-1. Prisma 스키마에 인덱스가 전혀 없음 ★ 최우선
- **현상**: `prisma/schema.prisma`에 `@@index`가 **0개**. `Post.authorId`, `FishingPoint.userId`, `MarketListing.sellerId/status/category`, `Like.postId`, `Comment.postId`, `Booking.userId`, `TournamentEntry(tournamentId,status)` 등 거의 모든 조회가 인덱스 없는 컬럼을 `where`/`orderBy`로 씁니다.
- **영향**: 데이터가 쌓일수록 풀스캔 → 피드·마켓·프로필·관리자 목록이 선형으로 느려짐.
- **권고**: 자주 필터/정렬하는 FK·상태 컬럼에 인덱스 추가. 예: `Post`에 `@@index([authorId, createdAt])`, `@@index([hidden, createdAt])`; `FishingPoint`에 `@@index([userId])`, `@@index([visibility])`; `MarketListing`에 `@@index([status])`, `@@index([category])`, `@@index([createdAt])`; `Comment/Like/Bookmark`에 `@@index([postId])`; `Booking`에 `@@index([userId, createdAt])`; `TournamentEntry`에 `@@index([tournamentId, status])`. 이후 `prisma migrate`.

### P0-2. 피드 쿼리의 좋아요/북마크 전체 로드 (N+1성 과다 조회)
- **현상**: `lib/queries.ts`의 `feedInclude`가 `likes: { select: { userId: true } }`, `bookmarks: { select: { userId: true } }`로 **각 게시글의 모든 좋아요·북마크 행**을 가져온 뒤, JS에서 `likes.some(userId===me)`로 내 상태를 계산합니다. 인기 게시글이면 한 게시글에 수백~수천 행을 통째로 전송.
- **권고**: 두 가지로 분리 — ① 카운트는 이미 `_count`로 충분, ② "내가 눌렀는지"는 현재 사용자만 조회하도록 `likes: { where: { userId }, select: { userId: true }, take: 1 }`처럼 **현재 사용자 필터**를 건다. 그러면 전송량이 게시글당 0~1행으로 줄어듦.

### P0-3. 모든 페이지가 `force-dynamic` — 캐싱 전무
- **현상**: `force-dynamic`이 **40개 파일**에 선언. 쇼핑 목록·예약 목록·대회 목록·상세처럼 사용자별 개인화가 적은 화면까지 매 요청 DB 히트.
- **권고**: 로그인 상태에 따라 내용이 바뀌지 않는 목록/상세(쇼핑, 예약 목록, 대회 목록, 상품/예약 상세)는 `export const revalidate = 60`(ISR) 또는 `unstable_cache` 적용. 개인화 요소(좋아요 여부 등)는 클라이언트에서 보강. 피드·마이처럼 진짜 개인화된 화면만 `force-dynamic` 유지.

### P0-4. `next/image` 미사용 — 이미지 최적화/지연로딩 없음
- **현상**: 코드 전체에서 `next/image` **0회**, 원시 `<img>` **33개**. 원격 이미지(unsplash·picsum·pravatar)가 리사이즈·webp 변환·우선순위 제어 없이 원본 그대로 로드. 일부는 `loading="lazy"`만 있고 다수는 그것도 없음.
- **권고**: 피드/마켓/쇼핑 썸네일을 `next/image`로 교체하고 `width/height`(또는 `fill` + `sizes`) 지정 → 자동 webp·반응형·지연로딩 + **CLS(레이아웃 점프) 제거**. `next.config.mjs`의 `remotePatterns`는 이미 등록돼 있어 전환 비용이 낮음.

### P1-5. 다수 목록 쿼리가 `take` 없이 무제한 조회
- **현상**: `findMany`에 `take`가 없는 곳이 다수 — `market/page.tsx`, `shop/page.tsx`, `reservations/page.tsx`, `products`, 그리고 거의 모든 `admin/*` 목록. 데이터가 늘면 한 번에 전부 끌어와 직렬화.
- **권고**: 목록에 페이지네이션(또는 `take: 30~50` + 무한 스크롤/더보기) 도입. 관리자 목록은 서버 페이지네이션 또는 커서 기반 권장.

### P1-6. 클라이언트 측 전량 필터링
- **현상**: `MarketList`/`FeedList`가 서버에서 받은 전체 배열을 클라이언트에서 검색·정렬·필터. 초기엔 OK지만 항목이 많아지면 전송량·메모리·렌더 비용 증가.
- **권고**: 검색/카테고리/정렬을 쿼리스트링 → 서버 쿼리로 위임(인덱스와 결합 시 시너지). 당장은 P0-5와 함께 take로 상한만 둬도 충분.

### P1-7. 마켓 상세: 읽기 요청마다 쓰기(조회수 +1)
- **현상**: `market/[id]/page.tsx`가 GET 렌더마다 `update viewCount increment` 실행 → 모든 조회가 쓰기 트랜잭션. 게다가 `force-dynamic`이라 캐시도 안 됨.
- **권고**: 조회수는 클라이언트 비동기 비콘(예: 별도 `POST /view`)으로 분리하거나 배치 집계. 정확도가 덜 중요하면 샘플링.

### P1-8. 렌더 블로킹 외부 폰트 CSS + 전역 Leaflet CSS
- **현상**: ① 루트 `<head>`에서 jsdelivr의 Pretendard CSS를 `<link rel="stylesheet">`로 동기 로드(렌더 블로킹, 외부 도메인 의존). ② `leaflet/dist/leaflet.css`를 **루트 레이아웃에서 전역 import** → 지도를 안 쓰는 모든 페이지에도 포함.
- **권고**: ① 폰트는 `next/font/local`로 self-host(자동 preload·FOUT 제어·외부 의존 제거). ② Leaflet CSS는 지도 컴포넌트(`MapCanvas`) 내부로 한정 import해 지도 라우트에서만 로드.

### P2-9. 폴링 기반 실시간 동기화
- **현상**: `MarketChatRoom`이 4초 간격 폴링, `MapScreen`도 주기적 갱신 패턴. 채팅방을 오래 열어두면 불필요한 요청 누적.
- **권고**: 단기적으로 백그라운드 탭에서 폴링 정지(`visibilitychange`)·지수 백오프. 중기적으로 SSE/WebSocket 검토.

### P2-10. 미사용/저활용 의존성 점검
- **현상**: `zod`가 설치돼 있으나 실제 import는 **1개 파일**뿐 — API 입력 검증에 거의 안 쓰임(대신 `b.x || null` 식 수동 파싱). 보안·안정성과 번들 모두에서 어정쩡.
- **권고**: API 입력 검증을 `zod`로 통일하거나(권장), 안 쓸 거면 제거. `date-fns`는 16개 파일에서 정상 사용 중(유지).

### P2-11. like 토글의 3쿼리
- **현상**: `/api/posts/[id]/like`가 `findUnique → delete/create → count` 3회. 트래픽이 몰리면 누적 비용.
- **권고**: `upsert`/`deleteMany` 1회 + 응답 카운트는 낙관적 UI(이미 적용됨)에 맡기고 주기적 정합으로 대체 가능. 우선순위는 낮음.

## 3. 디자인 / UI 정리 권고

### 3.1 색상 토큰 일관성 — 하드코딩 색상 50회+
- `tailwind.config.ts`에 `surface`(DEFAULT `#161616`, 200 `#1e1e1e` 등) 토큰이 정의돼 있는데도, 실제 코드엔 `#1e1e1e` **29회**, `#161616` **25회**, `#1a1a1a` 5회 등 **원시 hex가 직접** 박혀 있음(`#191919`, `#16243d` 같은 토큰에 없는 변형도 등장).
- **권고**: 모든 다크 표면을 `bg-surface`, `bg-surface-200` 등 토큰으로 치환. 한 곳(테마)만 바꾸면 전체가 따라오도록 단일화. 토큰에 없는 변형색은 토큰으로 흡수.

### 3.2 색상 네이밍이 의미와 반대 — `navy`가 사실상 흰색
- `navy-900 = #ffffff`(거의 흰 텍스트), `navy-50 = #222428`(어두운 면). 라이트 테마용 네이밍을 다크로 반전시켜 그대로 둔 흔적. 신규 작업자가 `text-navy-900`이 흰색인 걸 직관적으로 알 수 없어 실수 유발.
- **권고**: 시맨틱 네이밍으로 점진 리네임(`ink`/`fg`/`muted` 등) 또는 최소한 주석을 토큰 위에 표준화. 우선순위는 중간(대규모 치환이라 일정 잡고).

### 3.3 중복 컴포넌트
- `Chip`이 공용 `components/ui.tsx`와 `components/market/MarketList.tsx`에 **각각 정의**. 스타일이 미세하게 달라 칩 모양이 화면마다 어긋날 수 있음.
- 필터 상태도 `FeedHeader`가 자체 `active`를 들고, `FeedList`도 `filter`를 들어 **이중 소스**.
- **권고**: `Chip`은 `ui.tsx` 단일 출처로 통일하고 MarketList는 import만. 피드 필터는 상태를 한 곳(상위)에서만 관리.

### 3.4 빈 `src=""` 및 일관성 없는 이미지 폴백
- `shop/[id]`, `reservations/[id]` 운영자, 대회 리더보드 등에서 `src={p.imageUrl || ""}` 처럼 **빈 문자열**을 그대로 줌 → 깨진 이미지 아이콘/불필요한 요청 발생.
- 아바타 폴백도 화면마다 제각각: 어디선 `i.pravatar.cc`, 어디선 닉네임 첫 글자, 어디선 빈 src.
- **권고**: 공용 `<Avatar>`·`<Thumb>` 컴포넌트로 폴백(이니셜/플레이스홀더)을 1곳에서 처리. 빈 src 금지.

### 3.5 이미지 치수 미지정 → 레이아웃 점프(CLS)
- 다수 `<img>`에 `width/height` 없이 `aspect-*`만 의존하거나 그조차 없음. 로딩 중 점프 발생.
- **권고**: P0-4의 `next/image` 전환과 함께 명시적 치수 지정.

### 3.6 여백·간격 스케일 불규칙
- 동일 위계 헤더/패딩이 `px-3`, `px-3.5`, `px-4`, `px-5`로 혼재. 카드 패딩도 `p-2.5`~`p-4`가 섞임.
- **권고**: 페이지 좌우 패딩과 카드 패딩을 각각 1~2개 값으로 표준화(예: 본문 `px-4`, 카드 `p-4`). 공용 `PageHeader`/`Card`에 기본 패딩을 몰아 개별 지정 제거.

### 3.7 반응형 — 단일 760px 프레임 가정
- 전 화면이 `max-w-[760px]/[640px]` 중앙 프레임 + 우측 세로 네비 구조라 데스크톱 넓은 화면에서 좌우 여백이 크게 비어 보임. 마켓/쇼핑 그리드만 `md:grid-cols-3`로 늘어날 뿐.
- **권고**: 의도된 "모바일 프레임" 컨셉이면 현행 유지로 충분. 데스크톱 활용을 늘리려면 피드/마이에 보조 컬럼(추천·통계) 배치 검토. 우선순위 낮음(제품 방향 결정 사항).

### 3.8 파일 인코딩 불일치(부수 발견)
- 일부 마켓 페이지(`market/page.tsx`, `market/mine/page.tsx`, `market/chats/page.tsx`, `trip/page.tsx`)가 **UTF‑16**으로 저장돼 있어(grep이 binary로 인식) 도구·diff·검색에서 어긋날 수 있음.
- **권고**: 전 소스를 UTF‑8(LF)로 통일. 에디터/`.gitattributes`로 강제.

### 3.9 접근성 소소한 정리
- 일부 의미 있는 이미지의 `alt=""`(아바타·썸네일). 좋아요/저장 등 아이콘 버튼은 `aria-label`이 잘 붙어 있어 양호.
- **권고**: 콘텐츠 이미지엔 의미 있는 `alt`, 순수 장식만 `alt=""`.
